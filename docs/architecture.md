# Architecture & Conventions

## Package layout

```
src/
  session/            generic game session: bet, credits, base config
    videoslot/         video-slot-specific session, config, paytable, win calculation
      combinations/    reel sequences, spin generation, RNG, combination analysis
      linesdefinitions/  payline shapes
      linespatterns/     line matching rules (left-to-right, right-to-left, scattered)
      paytable/          payout tables
      wincalculator/     turns a symbols combination + paytable into wins
  simulation/          Monte Carlo RTP simulation, play strategies
    playstrategy/
  net/                 session → plain-data payload serializers
    videoslot/
```

`src/index.ts` is a flat barrel file re-exporting every source file. It's generated by `generate-barrels.js` (run as
part of `npm run build`) — never hand-edit it.

## Interface-per-role naming

Almost every concrete class is paired with one or more small interfaces named by the *role* they play, not the type
they belong to:

| Suffix | Role | Example |
|---|---|---|
| `*Describing` / `*Determining` | read-only query | `SymbolsSequenceDescribing`, `GameSessionStateDetermining` |
| `*Setting` | mutation | `SymbolsSequenceModifying`, `AvailableBetsSetting` |
| `*Representing` | full read+write composite — usually just `extends X, Y {}` with no members of its own | `VideoSlotConfigRepresenting` |
| `*Handling` | the operational surface of a session/simulation | `GameSessionHandling`, `VideoSlotSessionHandling` |
| `BuildableFrom*` / `ConvertableTo*` | plain-data import/export (`Array`, `Map`, or `Matrix` shaped) | `BuildableFromMatrix`, `ConvertableToMap` |

This is Interface Segregation in practice: a consumer that only needs to *read* a sequence depends on
`SymbolsSequenceDescribing`, not the mutable `SymbolsSequenceRepresenting`. It also means most "Representing"
interfaces are pure compositions and add no methods of their own — read the interfaces they extend to see the real
surface.

## Composition over inheritance, dependency injection via constructor defaults

There is effectively **no class inheritance** between domain classes anywhere in `src/`. Extended behavior is always
built by wrapping/composing an existing collaborator, typed against its interface, and injected through a
constructor default parameter:

```ts
export class VideoSlotSession implements VideoSlotSessionHandling {
    constructor(
        config: VideoSlotConfigRepresenting = new VideoSlotConfig(),
        combinationsGenerator: SymbolsCombinationsGenerating = new SymbolsCombinationsGenerator(config),
        winCalculator: VideoSlotWinCalculating = new VideoSlotWinCalculator(config),
        baseSession: GameSessionHandling = new GameSession(config),
    ) { /* ... */ }
}
```

Even `VideoSlotWithFreeGamesSession` — which reads like it should extend `VideoSlotSession` — instead holds a private
`baseSession: VideoSlotSessionHandling` and delegates most methods to it. Same pattern for
`VideoSlotWithFreeGamesConfig`, which wraps a private `VideoSlotConfig` rather than subclassing it.

Practical consequences:

- **Open/Closed extension points** are interface implementations, not subclasses. Want a different payline layout?
  Implement `LinesDefinitionsDescribing` (see [Paylines & Line Patterns](paylines-and-patterns.md)). Want a
  cryptographically secure RNG instead of the default `Math.random()`-based one? Pass a `SecureRandomNumberGenerator`
  into `SymbolsCombinationsGenerator`'s constructor (see [Reels & Symbol Sequences](reels-and-sequences.md)) — no
  existing class needs to change.
- **Every collaborator has a working default**, so `new VideoSlotSession()` with no arguments gives you a complete,
  playable game; you override only the pieces you need to customize.
- **Null-object pattern** shows up too: the generic `GameSession` depends on `WinAmountDetermining`, and defaults to
  `NoWinAmount` (always returns `0`) when no slot-specific win calculator is relevant.

## Serialization contracts are generic, not domain-specific

`BuildableFromArray`/`ConvertableToArray`, `BuildableFromMap`/`ConvertableToMap`, and
`BuildableFromMatrix`/`ConvertableToMatrix` (all in `src/session/`) are small generic interfaces mixed into unrelated
classes (`SymbolsSequence`, `SymbolsCombination`, `Paytable`, `CustomLinesDefinitions`, ...) to give each a uniform
plain-data round-trip:

```ts
interface BuildableFromMatrix<T = string> { fromMatrix(value: T[][], transposed?: boolean): this; }
interface ConvertableToMatrix<T = string> { toMatrix(transposed?: boolean): T[][]; }
```

All `Buildable*` methods return `this`, so they're chainable (`new SymbolsSequence().fromArray([...]).shuffle()`).
Concrete implementations deep-clone on both ends (via `JSON.parse(JSON.stringify(...))`), so the plain data you get
out — or pass in — is always decoupled from the object's internal state.
